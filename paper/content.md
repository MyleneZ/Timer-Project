# Tivvy Research Paper Paragraph Drafts

## 1.1 - Introduction to Problem (250 words)

Productivity is a long-standing problem for students, workers, and households because modern task management depends on attention, consistency, and the ability to transition between activities without losing track of time. Although timers and productivity applications are common, many of them are tied to smartphones, laptops, or cloud-connected smart speakers. These tools can be helpful, but they can also introduce the same distractions that they are supposed to reduce. Research on scheduling and procrastination shows that people often struggle to accurately manage their own time, even when they have freedom over their schedules [4]. Other work on productivity tools also suggests that reminders, time awareness, and structured task support can improve how people plan and complete work [3]. The problem is therefore not simply that people lack timers; rather, many existing timers are inconvenient, distracting, or dependent on online services. Voice interaction can make timers easier to use because users do not need to stop what they are doing, unlock a phone, or navigate an app. However, commercial voice assistants often raise privacy concerns because they may rely on microphones connected to cloud services. This affects students doing homework, people cooking or exercising, and users who want hands-free time management without sending voice data away from the device. In the long run, a more private and focused timer can help users maintain better work habits while avoiding unnecessary screen time. A dedicated offline voice timer addresses this gap by combining the convenience of speech control with the privacy and simplicity of a single-purpose physical device.

## 1.2 - Method Proposal (250 words)

Tivvy is a standalone, offline, voice-activated timer device that allows users to create, adjust, and stop task-related timers through spoken commands while receiving visual and audio feedback. The system is designed to solve the problem of distracting or privacy-sensitive productivity tools by removing the need for a smartphone, internet connection, or cloud-based voice assistant. Instead of asking the user to open an app, type a duration, or interact with a general-purpose device, Tivvy listens locally for a limited set of timer commands and responds through a compact hardware interface. The voice recognition component uses an edge device, the Nicla Voice chip, which pairs a microphone with a trained model for keyword spotting and command recognition. Because the model runs on the device, spoken commands can be processed without sending audio to an outside server. The Nicla communicates with the Qualia board, which handles the display, speaker output, and timer logic. Users can run up to three timers at once, view progress through circular timer rings, and hear sound effects or alarms when an event needs attention. This makes Tivvy effective because it focuses on one practical workflow: setting and monitoring time for everyday tasks. Compared with phone apps or smart speakers, the device is less distracting because it does not contain unrelated apps, notifications, or entertainment features. Compared with a basic kitchen timer, it is more flexible because it supports multiple named timers and hands-free changes. Its offline design also makes it more private and convenient in environments where internet access is unreliable or unwanted.

## 2.1 - Challenge A: Voice Recognition (100 words)

One major challenge would be building a reliable voice recognition system that still works entirely offline. Since Tivvy depends on spoken commands, the device would need to recognize words such as set, cancel, add, minus, stop, timer names, and time values accurately enough to be useful. This could become difficult because keyword spotting requires a strong training dataset, and users may speak at different volumes, speeds, or accents. The system also has to stay small enough to run on an edge device instead of relying on cloud processing. To address this, the project could use the Nicla Voice chip with a custom-trained model built from carefully collected command samples. Model testing could then be repeated in different environments so the final command set stays practical, private, and accurate.

## 2.2 - Challenge B: Enclosure Design and Parts Organization (100 words)

Another challenge would be designing the enclosure and organizing the internal hardware in a compact but usable way. Tivvy needs to include a display, speaker, battery or power input, a Qualia board, a Nicla Voice chip, and wiring between the boards. If these parts are placed inefficiently, the device could become too bulky, fragile, or difficult to assemble. The case also has to leave room for the screen to be visible, the speaker to be heard, and the microphone to capture commands clearly. To solve this, the project could use a 3D CAD program such as FreeCAD to model the case around the exact dimensions of each part. Several prototypes could be printed and revised until the device balances size, durability, and accessibility.

## 2.3 - Challenge C: Sound Design (100 words)

Sound design would also create important technical challenges because Tivvy uses audio feedback to tell users when commands or timers have completed. The speaker may have processing limitations, meaning that normal MP3 files may not be playable directly in the same way they would be on a phone or computer. The project would need a method for translating selected sound effects into raw data that the hardware can store and play back. It would also need clear rules for when each sound should play, such as timer completion, successful command recognition, or stopping an alarm. This could be resolved by gathering only the necessary sounds and creating a conversion script that automatically turns MP3 files into compatible data arrays. That approach would make future audio changes much easier to manage.

## 3.1B - System Overview (250 words)

Tivvy is structured around three major components: the voice recognition system, the hardware interface, and the software logic that manages timers. The voice recognition system begins with the Nicla Voice chip, which listens for a limited vocabulary of timer commands. This part of the system is responsible for turning spoken phrases into recognized tokens, such as set, add, minus, cancel, stop, timer names, and time values. The hardware component connects the Nicla Voice chip with the Qualia S3 RGB666 board, the display, the speaker, and the power system. The Qualia board controls the visible interface, audio playback, and the main behavior of the timers, while the enclosure keeps these parts in a portable form that can run through a wall connection or battery. The software component ties the system together by interpreting commands, creating timer objects, updating countdown values, rendering progress rings, and triggering alerts. From start to finish, the flow of the program begins when the user speaks a command near the device. The Nicla recognizes the command and sends structured information to the Qualia board through Bluetooth. The Qualia then parses that message to decide whether it should create a new timer, add or subtract time, cancel a specific timer, or stop an alarm. Once a timer is active, the display updates the timer name, remaining time, and progress ring so the user can understand the state of the device at a glance. The program was built using Arduino-based code, the Nicla Voice hardware, Edge Impulse-style dataset training for command recognition, Bluetooth communication, display rendering code, and custom enclosure design.

## 4.2A-4.2D - Experiment 2: Battery Life (280 words)

The second experimental blind spot is battery life under different environmental conditions and usage patterns. This is important because Tivvy is intended to be useful as either a wired or wireless device. If battery performance changes drastically depending on temperature, display activity, number of timers, or alarm behavior, users would need to understand those limitations before relying on the device for longer work sessions. The experiment should test temperature and device state separately. For the temperature portion, the device could be measured at room temperature, in a hot environment such as direct sunlight, and in a cold environment such as a refrigerator. Each run should begin at the same battery percentage, such as 100 percent or 80 percent, so the results can be compared fairly. The procedure would begin by recording the starting charge, setting a five-minute timer, allowing the timer to finish, and then recording the remaining battery percentage. For the device-state portion, the test should compare four conditions: sleep or idle state with no timer running, one active five-minute timer with the display on, three timers running at once, and an alarm state where the ringtone and visual alert are active. Each condition should run for a fixed length of time, such as ten or fifteen minutes, and each condition should be repeated two or three times. The final graph should show average battery loss per condition. It is expected that the alarm state and multiple-timer state would use the most power because the screen, logic, and speaker are more active. The idle state should use the least power, especially because the device auto-sleeps after three minutes without a timer. The most important factors will likely be display time, speaker use, command frequency, and the number of active timers.
